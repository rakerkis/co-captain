# Co-Captain Development Log — QR Token Scanner & Token Masking (v24)

## Overview

This log documents adding a QR code scanner for Canvas API token input on mobile and implementing token masking for security. These changes improve the mobile UX where long API tokens are difficult to type on a phone keyboard.

---

## v24 — QR Code Scanner for Canvas Token

**Brief:** Add camera-based QR code scanning to fill in the Canvas API token on mobile.

**Detailed:**
- **Problem:** On mobile devices (iOS/Android), the Canvas API token is a long random string that is extremely tedious to type manually on a phone keyboard. Users need an easier way to input it.
- **Solution:** Added a "Scan Token QR Code" button below the token input on native platforms. The workflow is:
  1. User copies their Canvas token on a computer
  2. User pastes it into any QR code generator website
  3. User taps the scan button in Co-Captain and points at the QR code
  4. Token auto-fills into the input field

### Plugin Selection

- **First attempt:** `@capacitor-mlkit/barcode-scanning` (Google ML Kit) — failed on iOS with `"barcode scanner plugin is not implemented in ios"`. The plugin distributes its iOS code via CocoaPods (`.podspec`), but this project uses Swift Package Manager (SPM). `cap sync` couldn't add it to the iOS `Package.swift`.
- **Final choice:** `@capacitor/barcode-scanner` (official Capacitor plugin) — uses `html5-qrcode` under the hood with native wrappers that support SPM. `cap sync` correctly added `CapacitorBarcodeScanner` to `ios/App/CapApp-SPM/Package.swift`.

### Android minSdkVersion Bump

- The `@capacitor/barcode-scanner` plugin depends on `io.ionic.libs:ionbarcode-android:2.0.1`, which requires `minSdkVersion 26`.
- The project was at `minSdkVersion 24`. Bumped to `26` in `android/variables.gradle`.
- API 26 = Android 8.0 (Oreo), released August 2017. This drops support for Android 7.x devices, which represent < 1% of active devices.

### iOS Camera Permission

- Added `NSCameraUsageDescription` to `ios/App/App/Info.plist`:
  > "Co-Captain needs camera access to scan QR codes for your Canvas API token."
- Without this key, iOS kills the app immediately when accessing the camera.

### UI Details

- The scan button only appears on native platforms (`Capacitor.isNativePlatform()`), hidden on web.
- Instructional text below the button tells users to generate a QR code from their token on a computer.
- Scan cancellation is handled silently (no error toast if user backs out).

### Files Modified:
- **`src/pages/Settings.tsx`** — Added `CapacitorBarcodeScanner` import, `handleScanToken()` handler, scan button with helper text (native only).
- **`ios/App/App/Info.plist`** — Added `NSCameraUsageDescription`.
- **`ios/App/CapApp-SPM/Package.swift`** — `CapacitorBarcodeScanner` dependency added by `cap sync`.
- **`android/variables.gradle`** — `minSdkVersion` 24 → 26.
- **`android/app/capacitor.build.gradle`** — Updated by `cap sync` to include barcode scanner plugin.
- **`android/capacitor.settings.gradle`** — Updated by `cap sync`.
- **`package.json`** / **`package-lock.json`** — Added `@capacitor/barcode-scanner` dependency, removed `@capacitor-mlkit/barcode-scanning`.

## v24 — Canvas Token Masking

**Brief:** Show the token in plain text only on first entry, then permanently mask it after saving.

**Detailed:**
- **Problem:** The Canvas API token is a sensitive credential. Previously it was stored and displayed as a `type="password"` input, but could still be revealed via browser dev tools or password managers. On mobile, `type="password"` also makes it hard to verify the token was entered correctly.
- **Solution:** Implemented a two-phase display:

  | State | What the user sees | Editable? |
  |-------|-------------------|-----------|
  | **New token** (just typed or scanned) | Full token in plain text (`type="text"`) | Yes |
  | **After saving** | First 6 characters + `••••••••••` | No (disabled input + "Change" button) |
  | **Returning to Settings** | Masked version loaded from storage | No (must click "Change" to enter new) |

- **How it works:**
  - `tokenIsNew` state tracks whether the user is actively entering a new token.
  - `maskedToken` state holds the display-only masked string (e.g. `abc123••••••••••`).
  - On mount, if a saved token exists, only the masked version is set — the full token stays in secure storage but is never put into React state for display.
  - After saving, the full token is written to secure storage, then immediately masked in the UI.
  - The "Change" button clears the masked display and lets the user enter a fresh token.
  - The full token remains in secure storage (Keychain on iOS, KeyStore on Android, localStorage on web) and is available for Canvas API calls.

### Files Modified:
- **`src/pages/Settings.tsx`** — Added `tokenIsNew` and `maskedToken` state, masked loading logic in `useEffect`, masking after save in `handleSave()`, conditional UI showing masked input with "Change" button vs editable input.

## v24 — Fix Google Token Auto-Refresh

**Brief:** Google Calendar stops working after 1 hour because the access token expires and refresh fails.

**Detailed:**
- **Symptom:** After ~1 hour, the home calendar shows no events and Settings shows "Google token expired. Please reconnect in Settings. [v23]". Tapping "Test Connection" confirms the failure.
- **Root Cause (refresh mechanism):** The previous `refreshGoogleToken()` method tried to call the Edge Function's `/events` endpoint, hoping it would auto-refresh the Google token server-side. But the Supabase Edge Function relay rejects the user's JWT (the same persistent 401 issue from v16-v22), so the refresh code never runs.
- **Root Cause (error version):** The `[v23]` version tags in Settings.tsx error messages were hardcoded strings, not updated when the version changed.

### Fix — New `/refresh` Edge Function Endpoint

Added a dedicated `/refresh` endpoint to the `google-calendar-auth` Edge Function that:
1. Accepts `user_id` and `refresh_token` in the POST body
2. Does **not** require user JWT auth (placed before the auth check in the EF)
3. Calls `https://oauth2.googleapis.com/token` with the `refresh_token`, `client_id`, and `client_secret` (server-side only)
4. Returns the new `access_token` and `expires_in`
5. Updates the `google_calendar_tokens` DB row with the fresh token

### Fix — Client-Side Refresh Flow

Rewrote `refreshGoogleToken()` in `googleCalendar.ts`:
- Reads the `refresh_token` from the DB via PostgREST (which works fine, unlike the EF relay)
- Calls the new `/refresh` EF endpoint with `Authorization: Bearer <anon_key>` (required by the Supabase relay — it needs a valid JWT, even for unauthenticated endpoints; the anon key is a valid JWT)
- Updates the DB with the new access token and expiry
- On success, continues with the calendar API call using the fresh token

### Fix — Shared Version Constant

- Created `src/lib/version.ts` exporting `APP_VERSION = "v24"`
- Updated `src/pages/Index.tsx` to use `{APP_VERSION}` instead of hardcoded `v24`
- Updated all `[v23]` error messages in `src/pages/Settings.tsx` to use template literals with `${APP_VERSION}`
- The JSX error display also uses `{APP_VERSION}` directly

### Debugging Detail

The initial `/refresh` implementation failed with a 401 because the `fetch()` call was missing the `Authorization` header. The Supabase Edge Function relay requires a valid JWT in the `Authorization` header for **all** requests — even if the Edge Function code itself doesn't check auth. The fix was to pass `Authorization: Bearer ${anonKey}`, since the project's anon key is a valid JWT that the relay accepts.

### Files Modified:
- **`src/integrations/googleCalendar.ts`** — Rewrote `refreshGoogleToken()` to use new `/refresh` endpoint. Improved error messages with actual failure reasons.
- **`src/pages/Settings.tsx`** — Imported `APP_VERSION`, replaced all hardcoded `[v23]` with `[${APP_VERSION}]`.
- **`src/pages/Index.tsx`** — Imported `APP_VERSION`, replaced hardcoded version string.
- **`src/lib/version.ts`** — New file, exports `APP_VERSION`.
- **`supabase/functions/google-calendar-auth/index.ts`** — Added `/refresh` POST endpoint before the auth check.

## Key Technical Lessons

1. **Not all Capacitor plugins support SPM.** `@capacitor-mlkit/barcode-scanning` only ships a CocoaPods `.podspec` — it won't work with SPM-based iOS projects. Always check the plugin's distribution format against your project setup.
2. **`@capacitor/barcode-scanner` uses `html5-qrcode`** with native wrappers, making it compatible with both CocoaPods and SPM.
3. **`NSCameraUsageDescription` is mandatory on iOS** for any camera access. Without it, iOS terminates the app immediately — no error dialog, no fallback.
4. **Token masking should happen at the display layer**, not the storage layer. Store the full token in secure storage; only mask what's shown in the UI. This keeps the API functional while preventing visual exposure.
5. **The Supabase Edge Function relay requires a valid JWT for ALL requests**, even endpoints that don't check auth. Use `Authorization: Bearer <anon_key>` for unauthenticated endpoints — the anon key is a valid JWT the relay accepts.
6. **Hardcoded version strings drift.** Use a shared constant (`APP_VERSION`) imported everywhere to keep debug tags in sync with the actual build.
