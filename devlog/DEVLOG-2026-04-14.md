# Co-Captain Development Log — Canvas Performance, GPA Grades, Google Token Refresh & Pull-to-Refresh (v26)

## Overview

This log documents four areas of work in v26: dramatically improving Canvas sync speed, fixing GPA grade display for observer (parent) accounts, hardening Google Calendar token refresh handling, and adding pull-to-refresh on the home page calendar.

---

## v26 — Fix: Canvas Sync Performance (1-2 min → ~5 sec)

**Brief:** Canvas data sync took 1-2 minutes on native. Reduced to ~5 seconds by switching to native HTTP and parallelizing API calls.

**Detailed:**
- **Problem:** Users reported Canvas sync taking 1-2 minutes before assignments and calendar events appeared. The "Test Connection" button (added in this release) responded quickly, but the full sync was painfully slow.
- **Root Causes:**
  1. **Proxy overhead on native:** Every Canvas API call was routed through a Supabase Edge Function CORS proxy (`canvas-proxy`), even on iOS/Android where CORS doesn't apply. Each call paid the penalty of: client → Supabase → Canvas → Supabase → client.
  2. **Sequential assignment fetching:** Assignments were fetched one course at a time in a `for` loop. With 7+ courses, this serialized 7 API round-trips.
  3. **WKWebView CORS enforcement:** Even when switching to direct `fetch()` on native, Capacitor's WKWebView still enforces CORS. Since Canvas doesn't return CORS headers, `fetch()` fails with a network error.

### Fix

**1. Use `CapacitorHttp` on native platforms:**

Replaced `fetch()` with `CapacitorHttp.get()` from `@capacitor/core` for Canvas API calls on native. `CapacitorHttp` uses the native HTTP stack (NSURLSession on iOS, OkHttp on Android), bypassing WKWebView's CORS enforcement entirely. This is the same approach already used by Outlook Calendar (`outlookCalendar.ts`).

```typescript
if (Capacitor.isNativePlatform()) {
  const nativeResp = await CapacitorHttp.get({
    url: `https://${baseUrl}/api/v1${path}`,
    headers: { Authorization: `Bearer ${token}` },
  });
  if (nativeResp.status >= 400) throw new Error(`Canvas API error ${nativeResp.status}`);
  return nativeResp.data;
}
```

**2. Parallel assignment fetching:**

Changed the sequential `for (const course of courses)` loop to `Promise.all(courses.map(...))`, fetching all course assignments concurrently.

**3. Better error messages on native:**

Added specific error messages for common failures:
- Network errors: "Network error connecting to {domain} — check your domain and internet connection."
- 401: "Invalid Canvas token — please check your API token in Settings."
- 403: "Canvas token does not have permission for this resource."

### Performance Impact

| Scenario (7 courses) | Before | After |
|-----------------------|--------|-------|
| Native (iOS/Android) | ~90s (7 sequential proxy calls) | ~5s (7 parallel direct calls) |
| Web | ~90s (7 sequential proxy calls) | ~15s (7 parallel proxy calls) |

### Files Modified:
- **`src/integrations/canvasApi.ts`** — Import `CapacitorHttp`; `canvasFetch()` uses native HTTP on Capacitor platforms; assignment fetching parallelized with `Promise.all`

## v26 — Fix: GPA Grades Not Showing (Observer Account Support)

**Brief:** GPA page showed N/A for all courses. Root cause: the Canvas account was an observer (parent) account, not a student account.

**Detailed:**
- **Problem:** GPA Calculator showed 7 courses but all grades were N/A, resulting in "Based on 0 graded courses" and GPA of N/A.
- **Investigation:** Added temporary debug output to the GPA page to inspect raw Canvas API responses. Discovered:
  1. The `/users/self/enrollments` endpoint returned 7 enrollments, all with `type: "ObserverEnrollment"` — not `"StudentEnrollment"`.
  2. The code filtered for `StudentEnrollment` only, so no grades were ever extracted.
  3. Observer enrollments don't carry grade data directly — grades belong to the observed student.

### Fix

Added observer account detection and a two-step grade lookup:

1. **Detect observer accounts:** Check if all enrollments have `type === "ObserverEnrollment"`.
2. **Find the observed student:** Extract `associated_user_id` from the first observer enrollment — this is the student being observed.
3. **Fetch the student's enrollments:** Call `/users/{observedUserId}/enrollments` to get the student's `StudentEnrollment` records, which contain `computed_current_grade` and `computed_current_score`.
4. **Fallback:** If `associated_user_id` is not available, fetch enrollments per-course and look for `StudentEnrollment` records there.

```typescript
const isObserver = allEnrollments.every((e) => e.type === "ObserverEnrollment");
if (isObserver) {
  const observedUserId = allEnrollments[0]?.associated_user_id;
  if (observedUserId) {
    const studentEnrollments = await canvasFetch(
      `/users/${observedUserId}/enrollments?per_page=100&state[]=active`, ...
    );
    // Extract grades from StudentEnrollment records
  }
}
```

### Debugging Approach

Since JavaScript `console.log` goes to Safari Web Inspector (not Xcode's console), debug logging was ineffective. Instead, added a temporary "Debug: Raw Enrollment Data" card to the GPA page that displayed raw API response data directly in the UI. This allowed the user to screenshot the data for diagnosis. Debug UI was removed after confirming grades worked.

### Files Modified:
- **`src/integrations/canvasApi.ts`** — Observer detection, observed student enrollment lookup, grade map construction with fallback
- **`src/pages/GPA.tsx`** — Temporary debug card (added then removed)

## v26 — Fix: Google Calendar Token Refresh (401 after 7 days)

**Brief:** Google Calendar showed "Token expired, refresh failed (401)" after several days. Fixed by handling token rotation and auto-disconnecting on failure.

**Detailed:**
- **Problem:** After ~7 days, Google Calendar stopped working with error: "Google token expired, refresh failed: Token refresh failed(401)[v25]". Reopening the app did not help.
- **Root Cause:** Google OAuth consent screen is in "Testing" mode, which limits refresh token lifetime to 7 days. After expiry, the refresh token is invalid and Google returns `invalid_grant` (401). Additionally, the app didn't handle refresh token rotation (where Google issues a new refresh token alongside the new access token).

### Fix (Client — `googleCalendar.ts`)

1. **Auto-disconnect on refresh failure:** When `refreshGoogleToken()` fails, the client now calls `this.disconnect()` to remove stale tokens from the DB. This ensures the user sees a clean "Connect" button in Settings instead of appearing connected but broken.
2. **Acknowledge token rotation:** If the Edge Function response includes `refresh_token_rotated: true`, the client logs it (the actual DB update happens server-side).

### Fix (Edge Function — `google-calendar-auth/index.ts`)

1. **Handle refresh token rotation:** If Google returns a new `refresh_token` in the refresh response, save it to the DB alongside the new access token.
2. **Better error logging:** Log `google_error` and `error_description` fields for debugging. Return `google_error` in the response so the client can display it.

### Files Modified:
- **`src/integrations/googleCalendar.ts`** — Auto-disconnect on refresh failure, token rotation acknowledgment
- **`supabase/functions/google-calendar-auth/index.ts`** — Save rotated refresh token, improved error logging and response

## v26 — Pull-to-Refresh on Home Page Calendar

**Brief:** Added slide-down-to-refresh gesture on the home page calendar for iOS/Android.

**Detailed:**
- **Problem:** No way to manually refresh calendar data on mobile without navigating away and back.
- **Solution:** Custom `usePullToRefresh` hook that handles touch gestures (start/move/end), with a visual RefreshCw spinner indicator. Triggers `queryClient.invalidateQueries()` to refresh all data sources (Canvas, Google Calendar, Outlook Calendar).

### Design Decisions

- **Native only:** Pull-to-refresh handlers are only attached on `Capacitor.isNativePlatform()` — not on web where it would conflict with browser scroll behavior.
- **Resistance curve:** Pull distance has a 0.5x multiplier with a 120px cap, giving a natural "rubber band" feel.
- **Threshold:** 80px pull distance required to trigger refresh.
- **Visual indicator:** Spinning `RefreshCw` icon with opacity and rotation based on pull distance, shown above the calendar content.

### Files Added:
- **`src/hooks/usePullToRefresh.ts`** — Custom hook: touch gesture tracking, pull distance with resistance, refreshing state management

### Files Modified:
- **`src/pages/Index.tsx`** — Import `usePullToRefresh` and `RefreshCw`; `handleRefresh` callback invalidates all queries; touch handlers on CardContent (native only); visual pull indicator

## Key Technical Lessons

1. **Capacitor's WKWebView enforces CORS even on native.** Using `fetch()` from a Capacitor app sends requests through WKWebView's network stack, which enforces CORS. APIs that don't return CORS headers (like Canvas) will fail with a network error. Use `CapacitorHttp` from `@capacitor/core` to route through the native HTTP stack instead.

2. **Canvas observer accounts have `ObserverEnrollment`, not `StudentEnrollment`.** Parent/guardian Canvas accounts are linked to students via `associated_user_id` on their enrollment objects. To get a student's grades from an observer account, you must first identify the observed student ID, then fetch that student's enrollments separately.

3. **JavaScript `console.log` in Capacitor goes to Safari Web Inspector, not Xcode.** When debugging on iOS, native (Swift/ObjC) logs appear in Xcode's console, but JS logs from the WebView appear in Safari's developer tools. For quick mobile debugging without connecting Safari, rendering debug data directly in the UI is more effective.

4. **Google OAuth "Testing" mode expires refresh tokens after 7 days.** Apps with consent screens in "Testing" status have a hard 7-day limit on refresh token lifetime. After expiry, the only recovery is re-authentication. Auto-disconnecting stale credentials gives users a clear path to reconnect instead of showing a cryptic error.

5. **Parallel API calls should be the default for independent data.** Sequential fetching of per-course data (assignments, enrollments) serializes network latency. `Promise.all` with per-item error handling (`try/catch` returning empty) gives the same resilience as a sequential loop but with dramatically better performance.
