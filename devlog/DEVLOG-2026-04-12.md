# Co-Captain Development Log — Outlook Calendar Integration & Sync Fixes (v25)

## Overview

This log documents adding Outlook Calendar sync to Co-Captain, fixing a false-positive "Sync Failed" toast that appeared on every home page refresh, and correcting version strings in error messages. Outlook Calendar follows the same no-login-required pattern as Canvas — users connect directly with their Microsoft account, no Co-Captain session needed.

---

## v25 — Outlook Calendar Integration

**Brief:** Add Outlook Calendar as a third calendar source, displaying events on the home page calendar alongside Canvas and Google Calendar. Requires no Co-Captain account.

**Detailed:**
- **Problem:** Co-Captain only showed Canvas assignments and Google Calendar events. Users with Microsoft/Outlook calendars had no way to see those events in-app.
- **Solution:** Full client-side OAuth 2.0 PKCE flow with Microsoft identity platform (Azure AD), storing tokens in `secureStorage` (Keychain on native, localStorage on web). Events are fetched directly from Microsoft Graph API, no server relay needed.

### Architecture Decision: Client-Side PKCE, No Supabase Session Required

The key design constraint was that Outlook Calendar must work **without** a Co-Captain login — the same model Canvas uses. This ruled out:
- Server-side OAuth (Edge Function callback + Supabase DB) — requires a logged-in user to associate tokens
- Supabase OAuth provider — ties the connection to the Co-Captain account

Instead, the final architecture mirrors the Canvas pattern:
1. Generate a PKCE code verifier + challenge client-side
2. Store the verifier in `secureStorage` (persists across the redirect)
3. Redirect to Microsoft's consent page
4. On callback, exchange the code + verifier for tokens client-side
5. Store tokens in `secureStorage` under `"outlook-calendar-tokens"`
6. Fetch events directly from Microsoft Graph API, refreshing the access token as needed

A Supabase Edge Function (`supabase/functions/outlook-calendar-auth/index.ts`) and DB migration (`outlook_calendar_tokens` table) were also created but are not used in the primary flow — kept for potential future server-side use.

### Azure App Registration

Required setup in Azure portal:
- App type: **Single-page application (SPA)** platform — critical for PKCE CORS support
- **Not** "Mobile and desktop applications" — that platform blocks cross-origin token exchange from a browser, causing `AADSTS90023` errors
- Redirect URIs registered: `http://localhost:8082`, `http://localhost:8080`, `co-captain://outlook-auth` (native)
- Scopes: `Calendars.Read offline_access openid profile email`
- Client ID: `495ad01e-4170-4a6d-aaaf-86de3240b8e8`

### Email Extraction Without Extra API Scopes

To show the connected account email in Settings, the implementation decodes the `id_token` JWT payload directly (no additional API call):

```typescript
const payload = JSON.parse(atob(tokens.id_token.split(".")[1].replace(/-/g, "+").replace(/_/g, "/")));
email = payload.email || payload.preferred_username || "";
```

This avoids needing `User.Read` scope and the `/me` Graph endpoint, which was returning 403 during development (the scope wasn't included).

### OAuth Callback Handling

Two paths depending on platform:

**Web:** `App.tsx` has a `useEffect` in `AppLayout` that checks for `?code=` in the URL on mount. If an Outlook PKCE verifier is present in localStorage (set by `outlookCalendar.connect()`), it treats the redirect as an Outlook callback, calls `outlookCalendar.handleCallback(code)`, dispatches `co-captain:outlook-auth-done`, and navigates to `/settings?outlook_auth=success`.

**Native (iOS/Android):** The existing `appUrlOpen` listener in `App.tsx` was extended to handle `co-captain://outlook-auth` deep links. Same token exchange + event dispatch flow.

### Settings UI

- Shows connected/disconnected state with connected email
- Error messages displayed **above** the connect button (not after)
- "Test Connection" button calls `getEvents()` to verify the token works end-to-end
- "Disconnect" clears tokens from `secureStorage`
- Error messages include `[${APP_VERSION}]` version tag for debugging

### Home Page Integration

Outlook events are merged into `allAssignments` in `Index.tsx` alongside Canvas and Google events:
- `course_id: "outlook-calendar"` for color assignment
- `isOutlookEvent: true` flag disables the completion checkbox
- Source icon: `public/outlook-calendar-icon.svg` (blue Microsoft envelope icon, rotated 90° clockwise)
- Link text: "Open in Outlook"

## v25 — Fix: "Sync Failed" False Positive Toast

**Brief:** Every home page refresh showed a "Sync Failed — 2 sources failed to sync" toast even though calendar events were loading correctly.

**Detailed:**
- **Root Cause:** The sync toast batching system added in the previous commit (069b898) uses `QueryCache.onError` to collect errors and fire a destructive toast. Canvas queries (`canvas-assignments`, `canvas-calendar-events`) throw their errors directly to the QueryCache when the Canvas API fails or credentials are missing — unlike Google and Outlook Calendar queries, which silently catch errors in their `queryFn` and return empty data.
- **Why events still showed:** Outlook and Google Calendar queries catch all errors internally and return `{ events: [], isConnected: false }` as a *success* from React Query's perspective. So calendar events from those sources rendered fine while the 2 Canvas query errors triggered the "2 sources failed" count.
- **Pattern inconsistency:** Google Calendar and Outlook Calendar hooks both wrap their `queryFn` in try/catch and return empty on failure. Canvas hooks did not, creating a inconsistency where Canvas errors surfaced as destructive toasts while other calendar source errors were silent.

### Fix

Wrapped both Canvas `queryFn` implementations with try/catch:

```typescript
// useCanvasAssignments — before
queryFn: async () => {
  const { assignments } = await fetchCanvasData(); // throws on any failure
  ...
}

// useCanvasAssignments — after
queryFn: async () => {
  try {
    const { assignments } = await fetchCanvasData();
    ...
    return { assignments: filteredAssignments };
  } catch {
    return { assignments: [] };
  }
}
```

Same pattern applied to `useCanvasCalendarEvents`. Canvas connectivity failures are now silent in the sync toast — consistent with how Google and Outlook handle failures.

### Files Modified:
- **`src/hooks/useCanvasAssignments.ts`** — Wrapped `queryFn` in try/catch, returns `{ assignments: [] }` on error
- **`src/hooks/useCanvasCalendarEvents.ts`** — Wrapped `queryFn` in try/catch, returns `[]` on error

## Key Technical Lessons

1. **Azure platform type matters for PKCE.** "Mobile and desktop applications" blocks cross-origin token exchange from browsers (`AADSTS90023`). For a web/SPA PKCE flow, URIs must be registered under "Single-page application" — even if the app also runs natively.
2. **JWT payloads contain profile data without extra scopes.** The `id_token` returned from Microsoft's token endpoint includes `email` and `preferred_username` in its payload. Decoding it client-side (`atob(token.split(".")[1])`) avoids needing `User.Read` and an extra Graph API round-trip.
3. **PKCE verifier must survive a redirect.** The code verifier is generated before the redirect and must be stored somewhere that persists across page navigation. `secureStorage` (or localStorage) is the right place — not React state, which is destroyed on redirect.
4. **Async state initialization can cause UI race conditions.** When a `useEffect` loads async data (like settings) and then calls a status check, both async operations must be sequenced — otherwise a slow settings load can overwrite a fast status check result.
5. **All calendar source queries should use the same error handling pattern.** Mixing throwing vs. silent-fail queries creates inconsistent UX: some sources silently return empty data while others fire destructive toasts for the same category of error (not connected / API failure). Pick one pattern and apply it consistently.

## v25.1 — Fix: Outlook OAuth & Event Sync on Native Platforms (iOS & Android)

**Brief:** Outlook Calendar connection succeeded on native (iOS and Android) but returned a 400 error during token exchange, and events never synced even after connecting. Same root causes affected both platforms.

**Detailed:**

### Issue 1: AADSTS90023 — Cross-Origin Token Redemption Blocked (400)

- **Root Cause:** On native platforms, Capacitor's WebView sends an `Origin: capacitor://localhost` header with `fetch()` requests. Microsoft's token endpoint rejects cross-origin token redemption for "Mobile and desktop" registered redirect URIs with error `AADSTS90023`.
- **Fix:** Replaced `fetch()` with `CapacitorHttp.post()` (from `@capacitor/core`) for the token exchange and token refresh calls on native platforms. `CapacitorHttp` uses the native HTTP stack (NSURLSession on iOS, OkHttp on Android), which does not send an `Origin` header, bypassing Microsoft's CORS enforcement entirely.
- **Web path unchanged:** Web continues to use `fetch()` since SPA-registered redirect URIs support cross-origin token exchange.

### Issue 2: Events Never Fetched — Query Gated on Supabase Auth

- **Root Cause:** `useOutlookCalendarEvents` hook had `enabled: isSignedIn`, where `isSignedIn` was derived from `supabase.auth.getSession()`. But Outlook Calendar uses standalone PKCE tokens stored in `secureStorage` — it has no dependency on Supabase auth. If the Supabase session was expired or absent, the query never fired.
- **Fix:** Replaced the Supabase auth check with `outlookCalendar.isAuthenticated()`, which checks whether Outlook tokens exist in `secureStorage`. Removed the unused `supabase` import from the hook.

### Issue 3: Graph API Fetch Also Blocked by CORS on Native

- **Root Cause:** Same `Origin: capacitor://localhost` issue affected the `getEvents()` call to Microsoft Graph API (`graph.microsoft.com`).
- **Fix:** `getEvents()` now uses `CapacitorHttp.get()` on native platforms, consistent with the token exchange fix.

### Issue 4: `outlook_auth=error` Not Handled in Settings

- **Root Cause:** When `App.tsx` navigated to `/settings?outlook_auth=error` after a failed token exchange, Settings.tsx only handled the `success` case. The `outlookLoading` state stayed `true` forever, leaving the button stuck in a "Connecting..." spinner.
- **Fix:** Added an `else if (outlookAuth === "error")` branch that resets `outlookLoading`, sets an error message, and shows a destructive toast. Also added `setOutlookLoading(false)` to the success branch as a safety net.

### Version Bump

- `APP_VERSION` bumped from `v24` to `v25` with comment: "Add Outlook Calendar sync (client-side PKCE OAuth + Microsoft Graph API)"

### Android Build Note

- `android/local.properties` needed `sdk.dir=/Users/user198/Library/Android/sdk` set manually — `ANDROID_HOME` was not in the shell environment.

### Files Modified:
- **`src/integrations/outlookCalendar.ts`** — Import `CapacitorHttp`; token exchange, token refresh, and `getEvents()` all use native HTTP on Capacitor platforms
- **`src/hooks/useOutlookCalendar.ts`** — Query gated on `outlookCalendar.isAuthenticated()` instead of Supabase session; removed `supabase` import
- **`src/pages/Settings.tsx`** — Handle `outlook_auth=error` param; add `setOutlookLoading(false)` to success path
- **`src/lib/version.ts`** — `v24` → `v25`

### Key Technical Lesson

6. **Capacitor WebView `fetch()` sends an `Origin` header that native HTTP does not.** When calling OAuth token endpoints or APIs that enforce CORS, use `CapacitorHttp` from `@capacitor/core` on native platforms. This routes through the native HTTP stack (NSURLSession on iOS, OkHttp on Android), which doesn't include browser-specific headers like `Origin` or `Referer`, avoiding CORS rejections entirely. This applies to both iOS and Android — the fix is platform-agnostic via `Capacitor.isNativePlatform()`.
