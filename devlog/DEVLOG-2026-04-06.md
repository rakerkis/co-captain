# Co-Captain Development Log — Google Calendar Integration

## Overview

This log documents the debugging journey to fix the Google Calendar OAuth flow and event syncing in the Co-Captain iOS Capacitor app. The core challenge was getting OAuth to work inside a native iOS WKWebView, handling custom URL scheme redirects, dealing with stale/invalid JWTs from the Supabase client, and ultimately bypassing the Supabase Edge Function relay entirely due to persistent 401 errors.

---

## v14 — Fix OAuth Freeze (SFSafariViewController)

**Brief:** App freezes after tapping "Continue" on Google consent screen.

**Detailed:**
- **Symptom:** User taps "Connect to Google" in Settings, Google consent screen appears in an in-app browser, user taps "Continue", and the app freezes — nothing happens, no redirect back to the app.
- **Root Cause:** The app was using `Browser.open()` from `@capacitor/browser`, which opens an `SFSafariViewController`. On iOS 14+, Apple introduced a security change where `SFSafariViewController` silently drops server-side HTTP 302 redirects to custom URL schemes (like `co-captain://`). The Supabase OAuth callback was doing a 302 redirect to `co-captain://settings?google_auth=success`, which was being silently swallowed.
- **Fix (Edge Function):** Modified the `google-calendar-auth` Edge Function's `/callback` route to return an HTML page with a JavaScript redirect (`window.location.replace(...)`) instead of a bare HTTP 302. JavaScript-level navigation in `SFSafariViewController` correctly triggers the OS URL handler and fires `appUrlOpen` in the app. Also included a tappable "Open App" link as a manual fallback.
- **Fix (Client):** Changed the OAuth launch from `Browser.open({ url: oauthData.url })` to `window.location.href = oauthData.url`. This leverages Capacitor 8's `WebViewDelegationHandler.swift`, which automatically intercepts top-level navigations to external URLs in WKWebView and opens them via `UIApplication.shared.open()`, launching native Safari instead of `SFSafariViewController`. Native Safari handles custom URL scheme redirects correctly.

## v15 — Fix Post-OAuth State Not Updating

**Brief:** Settings page doesn't show "Google Connected" after successful OAuth redirect.

**Detailed:**
- **Symptom:** After the OAuth flow completes and the app returns to the Settings page via `navigate("/settings?google_auth=success")`, the page still shows "Connect Google Account" instead of "Connected to Google Calendar".
- **Root Cause:** The Settings component was already mounted when the navigation happened. The `useEffect` that reads `location.search` didn't re-run because React Router's `useLocation()` hook wasn't triggering a re-render for the in-app navigation.
- **Fix:** Added multiple reactive mechanisms to catch the auth completion:
  1. **`useLocation()` reactive effect** — re-runs whenever `location.search` changes, checks for `google_auth=success` param.
  2. **CustomEvent dispatch** — `App.tsx`'s `appUrlOpen` handler dispatches a `co-captain:google-auth-done` CustomEvent after processing the OAuth callback. Settings listens for this event.
  3. **`appStateChange` fallback** — When the app returns to foreground (from Safari), waits 3 seconds then re-checks `isAuthenticated()` as a delayed fallback.
- **Also added:** `authLog()` persistent debug helper that writes timestamped entries to `localStorage` key `co-captain-auth-log` (up to 25 entries). Added a debug panel UI at the bottom of Settings page that shows these log entries.

## v16 — Attempt to Fix Invalid JWT (getUser + signOut)

**Brief:** Persistent `{"code":401,"message":"Invalid JWT"}` error when calling Edge Function.

**Detailed:**
- **Symptom:** After connecting Google Calendar, calling any Edge Function (status check, events, store) returns `{"code":401,"message":"Invalid JWT"}` from the Supabase Edge Function relay.
- **Attempted Fix:** Replaced `supabase.auth.getSession()` (which reads from local cache) with `supabase.auth.getUser()` (which validates the token server-side). Added `supabase.auth.signOut()` on failure to clear corrupted sessions.
- **Result:** Still failed. The 401 was coming from the Edge Function relay gateway (before user code runs), not from the application code inside the Edge Function.

## v17 — Attempt to Fix Invalid JWT (refreshSession + custom headers)

**Brief:** Added explicit `refreshSession()` and passed fresh token in headers.

**Detailed:**
- **Attempted Fix:** Added an explicit `supabase.auth.refreshSession()` call before invoking the Edge Function. Passed the freshly obtained `access_token` in the `headers` option to `supabase.functions.invoke()`.
- **Discovery:** `supabase.functions.invoke()` internally merges an `Authorization: Bearer <anon_key>` header as a base header. Custom headers passed via the `headers` option don't reliably override this internal header due to merge order. The Edge Function relay was receiving the anon key instead of the user's JWT.
- **Result:** Still failed — same 401 error.
- **Also added:** Version tagging in error messages (e.g., `[v17]`) to track which build was running.

## v18 — Bypass supabase.functions.invoke() with raw fetch()

**Brief:** Replaced `supabase.functions.invoke()` with raw `fetch()` using explicit headers.

**Detailed:**
- **Attempted Fix:** Completely bypassed `supabase.functions.invoke()` and used raw `fetch()` with explicit `Authorization: Bearer ${freshToken}` and `apikey: ${anonKey}` headers for the `connect()` method's Edge Function call.
- **Key Discovery:** Even with raw `fetch()` and a freshly refreshed token, the call still returned 401 on Xcode builds — but `npx cap run ios` (which does a fresh app install, clearing localStorage) worked fine. This confirmed the issue was **stale session data in localStorage** from previous Xcode simulator runs. Xcode re-uses the simulator's app data between builds, while `npx cap run ios` installs a clean copy.
- **Result:** Partial success — works on clean installs, still fails on Xcode re-runs with leftover state.

## v19 — 401 Fallback to Fresh OAuth

**Brief:** If Edge Function returns 401, automatically fall back to a fresh sign-in.

**Detailed:**
- **Fix:** Added a 401 fallback in the `connect()` method. If the Edge Function relay returns a 401 status despite `getUser()` and `refreshSession()` succeeding, the code automatically calls `startFreshOAuth()` — which signs the user out completely (`supabase.auth.signOut()`), clears the corrupted session, and initiates a brand new `signInWithOAuth` flow with Google.
- **Result:** This fixed the JWT error. The flow now self-heals: corrupted session → 401 from relay → automatic sign-out + fresh OAuth → clean session → success.

## v20 — Fix URL Scheme Mismatch

**Brief:** `appUrlOpen` handler not firing because of `co-captain:` vs `co-captain://` mismatch.

**Detailed:**
- **Symptom:** After fixing the 401 error, the Xcode console showed the deep link URL was `co-captain:?code=...` (no `//` after the colon), but the code was checking `url.startsWith("co-captain://")` which didn't match.
- **Root Cause:** iOS strips the `//` from custom URL scheme redirects in certain contexts. The URL `co-captain://settings?google_auth=success` becomes `co-captain:?code=...` when delivered through `UIApplication.shared.open()`.
- **Fix:** Changed the scheme check from `url.startsWith("co-captain://")` to `url.startsWith("co-captain:")`, which matches both `co-captain://...` and `co-captain:...`.
- **Result:** Settings page now correctly shows "Connected to Google Calendar" with the user's email after OAuth. The `appUrlOpen` handler fires, the PKCE code exchange completes, and the `co-captain:google-auth-done` CustomEvent is dispatched.

## v21 — Convert getEvents/isAuthenticated/disconnect to raw fetch()

**Brief:** Calendar events not appearing despite successful connection.

**Detailed:**
- **Symptom:** Settings shows "Connected to Google Calendar" with email, but the Home calendar page shows no events. User expects events on April 3 & 4.
- **Root Cause:** Only `connect()` had been converted to raw `fetch()` with explicit headers. The `isAuthenticated()`, `getEvents()`, and `disconnect()` methods were still using `supabase.functions.invoke()`, which had the same stale JWT problem.
- **Fix:** Converted all three methods from `supabase.functions.invoke()` to raw `fetch()` with explicit `Authorization: Bearer ${session.access_token}` and `apikey` headers.
- **Also fixed:** Added `co-captain:google-auth-done` CustomEvent listener in the `useGoogleCalendarEvents` hook. When Google auth completes, the hook now invalidates the `["google-calendar-events"]` React Query cache and sets `isSignedIn = true`, triggering a refetch. Also invalidates on `onAuthStateChange` with a new session.
- **Result:** Still got 401 — the raw `fetch()` was using `getSession()` which returns cached/stale tokens.

## v22 — Switch from getSession() to refreshSession()

**Brief:** Replace all `getSession()` calls with `refreshSession()` for fresh tokens.

**Detailed:**
- **Symptom:** Same 401 error on "Test Connection" in Settings.
- **Root Cause:** `isAuthenticated()`, `getEvents()`, and `disconnect()` were all calling `supabase.auth.getSession()` which returns **cached** session data from localStorage. On the iOS simulator with leftover state from previous builds, this cache contains an expired/invalid JWT. `refreshSession()` forces a server-side token refresh and returns a guaranteed-fresh JWT.
- **Fix:** Replaced `supabase.auth.getSession()` with `supabase.auth.refreshSession()` in all three methods (`isAuthenticated()`, `getEvents()`, `disconnect()`), matching the pattern already working in `connect()`.
- **Result:** Still failed — the fundamental problem was that the Edge Function relay itself was broken for this project/deployment. Even fresh, valid JWTs were being rejected.

## v23 — Bypass Edge Function Relay Entirely (Direct DB + Direct Google API)

**Brief:** Complete architectural change — bypass Edge Function for all read/write/delete operations.

**Detailed:**
- **Symptom:** Every Edge Function call returns `{"code":401,"message":"Invalid JWT"}` regardless of token freshness. The Supabase Edge Function relay gateway rejects the JWT before user code in the Edge Function even runs. Settings shows "Connected" initially (from the OAuth redirect params) but reverts to "Unconnected" when `isAuthenticated()` fails. Test Connection shows the same 401 error.
- **Root Cause (final diagnosis):** The Edge Function relay is persistently rejecting all user JWTs for this project. The `store` call in `App.tsx` after PKCE exchange was also using `supabase.functions.invoke()`, meaning **Google tokens were never actually saved to the database**. The UI showed "Connected" because it was reading from the URL params (`google_auth=success`), not from the database.
- **Fix — bypass the Edge Function relay entirely for all operations:**

  | Operation | Before (broken) | After (v23) |
  |---|---|---|
  | **Store tokens** (App.tsx after PKCE) | `supabase.functions.invoke("google-calendar-auth/store")` → 401 silently caught | `supabase.from("google_calendar_tokens").upsert(...)` via PostgREST |
  | **Check connected** (`isAuthenticated()`) | Raw `fetch()` to Edge Function `/status` → 401 | `supabase.from("google_calendar_tokens").select("user_id").eq("user_id", ...).maybeSingle()` |
  | **Fetch events** (`getEvents()`) | Raw `fetch()` to Edge Function `/events` → 401 | Read tokens from DB → call `https://www.googleapis.com/calendar/v3/calendars/primary/events` directly |
  | **Disconnect** (`disconnect()`) | Raw `fetch()` to Edge Function `/disconnect` → 401 | `supabase.from("google_calendar_tokens").delete().eq("user_id", ...)` |

- **New `storeTokens()` method:** Added a public `storeTokens(accessToken, refreshToken)` method on `GoogleCalendarService` that writes directly to the `google_calendar_tokens` table via Supabase PostgREST. Called from `App.tsx` in both the PKCE and implicit auth paths.
- **Direct Google Calendar API calls:** `getEvents()` now reads the stored Google access token from the database and calls the Google Calendar API directly (`/calendar/v3/calendars/primary/events`). If the Google token is expired, it attempts a refresh via the Edge Function as a fallback (since the EF has the Google `client_secret` needed for the refresh token grant).
- **Time range fix:** Changed `timeMin` from `new Date()` (current time, excluding past events) to `Date.now() - 30 days`, so recent past events (like the user's April 3-4 events) are included.
- **RLS compatibility:** The `google_calendar_tokens` table has Row Level Security enabled with policies for `auth.uid() = user_id` on SELECT, INSERT, UPDATE, and DELETE. The Supabase JS client automatically includes the user's JWT in PostgREST requests, so RLS works correctly without going through the Edge Function.
- **App.tsx imports:** Changed from dynamic `import()` to static import of `googleCalendar` to eliminate Vite warnings about mixed static/dynamic imports of the same module.

### Files Modified in v23:
1. **`src/integrations/googleCalendar.ts`** — Complete rewrite of `isAuthenticated()`, `disconnect()`, `getEvents()`. Added `storeTokens()` and `refreshGoogleToken()` helper. Only `connect()` still uses the Edge Function (for the initial OAuth URL generation, which requires the Google `client_secret`).
2. **`src/App.tsx`** — Replaced `supabase.functions.invoke("google-calendar-auth/store")` with `googleCalendar.storeTokens()` in both PKCE and implicit auth paths. Added static import of `googleCalendar`.
3. **`src/pages/Index.tsx`** — Version bump to v23.
4. **`src/pages/Settings.tsx`** — Error tag version bump to [v23].

### Note for User:
Since the old `store` calls via Edge Function were silently failing, Google tokens were never actually saved to the database in previous versions. The user needs to **disconnect and reconnect** Google Calendar in Settings to save fresh tokens using the new direct-DB method. After that, events should appear on the calendar.

## Key Technical Lessons

1. **`SFSafariViewController` drops custom scheme 302 redirects on iOS 14+.** Use JavaScript-level redirects or native Safari.
2. **Capacitor 8's `WebViewDelegationHandler.swift`** intercepts external URL navigations and opens them via `UIApplication.shared.open()`. Use `window.location.href` to trigger this.
3. **iOS may strip `//` from custom URL schemes.** Always check with `startsWith("scheme:")` not `startsWith("scheme://")`.
4. **`supabase.functions.invoke()` merges its own `Authorization: Bearer <anon_key>` header** and custom headers don't reliably override it. Use raw `fetch()` for full header control.
5. **`supabase.auth.getSession()` returns cached data from localStorage.** On iOS simulators that persist data between Xcode builds, this cache can contain stale/expired tokens. Use `getUser()` for server-side validation or `refreshSession()` for a fresh token.
6. **The Supabase Edge Function relay validates JWTs before user code runs.** If the relay rejects the token, your Edge Function code never executes. When this validation is unreliable, bypass the relay by talking directly to PostgREST (Supabase DB client) and external APIs.
7. **`npx cap run ios` installs a fresh app (clean localStorage)** while Xcode Run re-uses the existing app data on the simulator. This difference can mask auth bugs.
